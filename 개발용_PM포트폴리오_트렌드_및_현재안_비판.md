# PM/서비스 기획 포트폴리오 트렌드와 현재안 비판

작성일: 2026-06-17  
목적: 인터엑스 사전과제 웹 포트폴리오를 채용담당자/인사담당자의 시선에서 다시 설계하기 위한 기준 정리.

## 1. 최근 PM/서비스 기획 포트폴리오 트렌드

최근 PM/서비스 기획 포트폴리오 관련 글과 멘토링 시장에서 공통적으로 보이는 흐름은 다음과 같다.

### 1. 지원 기업 맞춤형 구성

포트폴리오는 모든 경험을 펼쳐놓는 문서가 아니라, 지원 기업이 보고 싶은 역량을 빠르게 판별하도록 구성해야 한다.

근거:

- PM 포트폴리오 팁 글은 기업이 가장 먼저 보는 것이 `입사 후 시킬 일과 경험의 유사성`이라고 설명한다.
- 2025 PM/PO 멘토링 소개에서도 기업 분석 전략, 스토리텔링 고도화, 기업 관점 최종 검토를 핵심 프로세스로 둔다.

현재안 문제:

- 현재 웹은 인터엑스 맥락을 반영했지만, `왜 이 프로젝트가 인터엑스 제품군과 직접 연결되는가`가 아직 한 단계 약하다.
- Dimetric, ireh link, StockStalker가 좋은 사례임에도 각각의 인터엑스 제품군과 대응되는 시각 장치가 부족하다.

개선 방향:

- `INTERX가 만드는 것`과 `김기범이 이미 만든 것`을 1:1 매칭하는 섹션을 추가한다.
- 단순 역량 키워드 대신 `Gen.AI`, `Admin/Dashboard`, `Data Space`, `A-to-Z Launch`처럼 공고/회사 언어로 재번역한다.

### 2. WHY-HOW-WHAT 흐름

PM 포트폴리오는 문제 정의, 의사결정, 실행 결과가 이어져야 한다. 프로젝트 설명은 `무엇을 했다`보다 `왜 그렇게 풀었고 어떤 결과가 나왔는가`가 중요하다.

현재안 문제:

- 현재 Case 섹션은 화면 이미지는 강하지만, 문제-판단-산출물-결과의 흐름이 압축되어 있다.
- 특히 ireh link는 클라이언트 요구사항이 러프했던 상황, 레퍼런스 수집, PRD/IA/SB/Figma/QA로 구체화한 과정이 더 보여야 한다.

개선 방향:

- 각 주요 Case는 `Problem → Decision → System → Evidence` 4단계로 고정한다.
- 긴 글 대신 타임라인, 파이프라인, 수치 카드, 도식으로 보여준다.

### 3. 숫자와 결과 중심

경력/주니어 포트폴리오일수록 숫자와 결과가 중요하다. 숫자는 두리뭉실한 자기평가를 줄이고, 실제로 해본 사람이라는 인상을 만든다.

현재안 문제:

- StockStalker 1,000명/12,000조회는 잘 보이지만, 자투리/테이스팃/사주중독/Dimetric 수치가 화면에서 약하다.
- 공고의 우대사항인 초기 스타트업 경험, 데이터 추출/분석 가능성이 분리되어 보이지 않는다.

개선 방향:

- 모든 수치는 차트화한다.
- 숫자 목록이 아니라 `운영 전환`, `출시`, `데이터`, `초기창업` 네 축의 metric field로 묶는다.

### 4. 산출물과 협업 증거

서비스 기획자는 주요 산출물(PRD, 요구사항 정의서, 플로우차트, 와이어프레임/프로토타입, QA 시나리오)을 통해 협업 가능성을 증명한다.

현재안 문제:

- 현재 웹은 PRD/IA/Figma/QA 시트를 말로만 언급한다.
- 실제 산출물 이미지가 없더라도 `산출물의 역할`을 구조로 보여줄 필요가 있다.

개선 방향:

- ireh link 섹션에 `Rough request → Reference → PRD → IA/SB → Figma → QA sheet → Release` 공정도를 넣는다.
- 이 공정도는 제조 공정과 유사한 시각 언어로 만들어 인터엑스와 맞춘다.

### 5. 웹 포트폴리오는 장표가 아니라 경험이어야 한다

사전과제에서 `단순 키워드 나열 및 텍스트 위주 지양`이 명시되어 있다. 웹으로 만든다면 PDF 장표를 그대로 옮기는 것이 아니라, 스크롤을 발표 흐름으로 만들어야 한다.

현재안 문제:

- IntegratedBio식 sticky hero와 큰 타이포는 들어갔지만, 아래 섹션은 아직 상대적으로 일반적인 사례 나열에 가깝다.
- `수치 하나하나를 벤치마킹`한다는 목표에 비해 데이터 시각화 밀도가 부족하다.

개선 방향:

- IntegratedBio의 구성 원리를 다음처럼 적용한다.
  - 긴 sticky hero: 후보자의 핵심 명제를 3단계로 증명
  - platform grid: 인터엑스 제품군과 후보 경험 매칭
  - card reveal: 세 가지 핵심 역량을 장면처럼 전환
  - newsroom/evidence: 실제 화면과 수치 증거를 카드로 편집
  - closing: 제조 경험 부족을 솔직하게 인정하되, Jr. 포지션 요구와 정확히 맞춘다.

## 2. 현재 웹 포트폴리오 비판

### 좋은 점

- 첫 화면에서 `기획에서 멈추지 않고 제품으로 출고`라는 강한 명제가 있다.
- IntegratedBio식 sticky hero, 01/03 진행, 대형 타이포, 이미지 레이어 전환이 이미 들어갔다.
- Dimetric, ireh link, StockStalker라는 우선순위는 인터엑스에 맞다.
- 채팅 모달은 AI Native 포트폴리오의 차별 요소가 될 수 있다.

### 부족한 점

| 문제 | 왜 약한가 | 보완 |
|---|---|---|
| 회사/공고 매칭이 설명문에 머문다 | 채용담당자는 긴 설명을 읽기 전에 적합성 판단을 끝낸다 | `INTERX Product Map x My Proof` 시각 섹션 추가 |
| 수치가 카드 문장에 묻힌다 | 사전과제 요구가 도표/차트 활용이다 | 모든 수치를 bar, radial, timeline, sparkline 형태로 시각화 |
| ireh link의 PM 범위가 약하게 보인다 | 실제 핵심 경험인데 화면 한두 장으로만 소비된다 | rough request → launch 공정도 추가 |
| Dimetric의 AI Native 가치가 충분히 크지 않다 | 단독 개발 프로젝트의 기술/제품 깊이가 강점인데 현재는 설명 한 문단 | 자연어 → patch → compile → snapshot → export flow를 hero급으로 시각화 |
| 초기 창업 경험이 부록처럼 보인다 | 우대사항인 “안 되면 되게 하는” 성장 과정과 직접 연결된다 | Jaturi/Tastit를 `operator proof`로 수치화 |
| 웹다운 몰입감은 첫 화면에 집중되어 있다 | 아래로 갈수록 장표형 카드가 남아 있다 | 섹션마다 scroll progress와 이미지/차트 변화를 붙인다 |

## 3. 채용담당자 관점의 판단 흐름

채용담당자는 이 포트폴리오를 다음 질문으로 본다.

1. 이 사람은 Jr. PO 필수 경력 1년 이상을 충족하는가?
2. 제조 도메인 부족을 보완할 만큼 빠르게 구조화하고 실행할 수 있는가?
3. AI Native가 말뿐이 아니라 실제 제품 제작 능력인가?
4. B2B/Admin/Dashboard 경험이 있는가?
5. 개발자·디자이너와 협업할 수 있는 산출물과 프로세스를 갖췄는가?
6. 초기 스타트업처럼 불확실한 상황에서 직접 부딪혀본 사람인가?

따라서 웹의 흐름은 다음 판단에 맞춰야 한다.

- 10초: 핵심 명제와 인터엑스 적합성을 인지
- 30초: Dimetric/ireh link/StockStalker 3대 증거 확인
- 1분: 수치와 산출물로 진짜 해본 사람인지 확인
- 3분: 질문이 생기면 채팅으로 프로젝트 상세 확인

## 4. 설계 원칙

- 과장하지 않는다. 숫자는 이력서 PDF에 있는 값만 사용한다.
- `AI Native`는 도구명 나열이 아니라 결과물과 프로세스로 보여준다.
- `제조 경험 부족`은 숨기지 않고, 공고의 Jr. 포지션 문장과 연결해 정면 돌파한다.
- `개발 베이스/데이터 추출·분석` 우대사항은 StockStalker와 Dimetric으로 보여준다.
- `초기 멤버로 안 되면 되게 한 경험`은 자투리/테이스팃의 방문 설문, PoC, 정부지원, 매출 수치로 증명한다.
